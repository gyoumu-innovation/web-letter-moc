# 詳細フェーズ計画書

## フェーズ概要

| フェーズ | 期間目安 | 主要成果物 |
|---|---|---|
| Phase 1：設計完了 | 〜2週間 | 設計書全件確定・技術スタック決定 |
| Phase 2：環境構築 | 〜1週間 | インフラ・開発環境セットアップ完了 |
| Phase 3：開発 | 〜6週間 | 全機能実装完了 |
| Phase 4：テスト | 〜2週間 | テスト完了・不具合修正 |
| Phase 5：リリース準備 | 〜1週間 | 本番環境構築・リリース |

| API設計書作成 | レビューOK |

## Phase 3：開発（〜6週間）

### 開発優先順位

| 優先度 | 機能グループ | 理由 |
|---|---|---|
| 高（先行） | 認証・ログイン・権限制御（RBAC基盤） | 他機能の土台になるため最初に実装 |
| 高 | 注文（入稿）・受注管理・差戻し | コア業務フローのため優先 |
| 中 | ユーザー管理・企業管理 | 認証基盤が完成してから着手 |
| 中 | 通知・メール送信（SES連携） | 受注管理完了後に統合 |
| 低 | 代理ログイン・操作履歴閲覧 | コア機能完了後に対応 |

### 外部IF連携方針

- ろぼすけ連携の仕様確定まではモックで先行開発
- SES連携はステージング環境で動作確認後に本番適用

---

## リスクと対策

| リスク | 対策 |
|---|---|
| 技術スタック未定による設計変更 | Phase 1の2週間以内に確定させる |
| ろぼすけ外部IF仕様確定の遅れ | モックで先行開発・並走で仕様確定 |
| テストで不具合が多発した場合 | Phase 4のバッファで吸収。超過する場合はリリース延期を検討 |
| SESサンドボックス解除の遅れ | Phase 2開始時点で申請。未承認の場合はSendGridで代替 |
