# インフラ環境設計書

## 概要

ロボットレターWEB発注システムのインフラ環境構成を定義する。

---

## インフラ選定比較：AWS vs さくらインターネット

| 項目 | AWS | さくらインターネット |
|---|---|---|
| DBマネージド | RDS（自動バックアップ・フェイルオーバー） | DBサーバ（手動管理） |
| ファイルストレージ | S3（送付リスト・画像の保存） | オブジェクトストレージ（別途設定） |
| スケーラビリティ | 高（オートスケーリング可） | 中（プラン変更で対応） |
| コスト | 従量課金（変動あり） | 固定料金（見積もりやすい） |
| 国内リージョン | 東京・大阪 | 東京・大阪・石狩 |
| 監視連携（Datadog） | 容易（CloudWatch経由） | 可能（エージェント導入） |
| 構築難易度 | 中〜高 | 低〜中 |
| 障害対応 | マネージドサービスで対応しやすい | 自前運用が多くなる |

### 推奨：AWS

**理由：**
- 送付リスト・サンプル文面のファイルアップロードにS3が最適
- RDSで自動バックアップ・マルチAZ構成によるDB冗長化が容易
- Datadogとの連携がCloudWatch経由でスムーズ
- SESでのメール送信が既存構成に統合しやすい

**さくらが適するケース：**
- コストを固定費で管理したい
- インフラ担当がさくら環境に習熟している
- AWSの設定工数を省きたい

## 環境構成

### 環境分け

| 環境 | 用途 | 構成 |
|---|---|---|
| 開発（dev） | 開発者のローカル動作確認 | ローカル（Docker） |
| ステージング（stg） | テスト・受け入れ確認 | 本番同等構成（スペックは縮小） |
| 本番（prod） | 実サービス稼働 | フルスペック |

### AWS構成（推奨案）

| コンポーネント | サービス | 用途 |
|---|---|---|
| Webサーバ | EC2（t3.small〜medium） | アプリケーションサーバ |
| DB | RDS（MySQL 8.0、db.t3.small〜） | メインDB |
| ファイルストレージ | S3 | 送付リスト・サンプル文面・帳票 |
| CDN | CloudFront（任意） | 静的ファイル配信 |
| メール送信 | SES or SendGrid | 通知メール送信 |
| ロードバランサ | ALB | HTTPS終端・振り分け |
| DNS | Route 53 | ドメイン管理 |

### さくらインターネット構成（代替案）

| コンポーネント | サービス | 用途 |
|---|---|---|
| Webサーバ | さくらのクラウド（2コア/4GB〜） | アプリケーションサーバ |
| DB | さくらのクラウド DBサーバ | メインDB |
| ファイルストレージ | さくらのオブジェクトストレージ | 送付リスト・サンプル文面 |
| メール送信 | SendGrid（外部サービス） | 通知メール送信 |
| SSL証明書 | Let's Encrypt | HTTPS対応 |

---

## ネットワーク構成（AWS）

```
インターネット
  ↓
Route 53（DNS）
  ↓
ALB（HTTPS:443）
  ↓
EC2（アプリサーバ / VPC内）  ←→  S3（ファイルストレージ）
  ↓
RDS（DB / プライベートサブネット）
```

---

## セキュリティ

| 項目 | 設定 |
|---|---|
| HTTPS | ACM証明書 + ALBでTLS終端 |
| DBアクセス | VPC内のみ（パブリックアクセス無効） |
| S3 | バケットポリシーでEC2のみアクセス許可 |
| SSHアクセス | Systems Manager Session Manager経由（直接SSH不可） |
| WAF | AWS WAF（将来対応） |

---

## 備考

- ドメインは別途取得が必要
- SESはサンドボックス解除の申請が本番リリース前に必要
- ステージング・本番でDBの接続情報は環境変数で管理（コードに埋め込み禁止）
