# システム運用設計書

## 概要

ロボットレターWEB発注システムの本番運用における、メンテナンス・ログ・バックアップ・監視の方針を定義する。

---

## メンテナンス

### 定期メンテナンスウィンドウ

| 項目 | 設定 |
|---|---|
| 実施タイミング | 毎週日曜 AM 2:00〜4:00（JST） |
| 通知タイミング | 3日前にシステム内通知・メール送信 |
| メンテナンス画面 | 503ページ表示 |
| 対象作業 | DBマイグレーション・パッケージ更新・証明書更新等 |

### 緊急メンテナンス

- 重大障害発生時はメンテナンスモードに即時切り替え
- 対象企業へ案内メール送信

---

## ログ

### ログ種別と保存方針

| 種別 | 内容 | 保存先 | 保存期間 |
|---|---|---|---|
| アクセスログ | HTTPリクエスト・レスポンス | CloudWatch Logs | 90日 |
| アプリケーションログ | エラー・警告・情報 | CloudWatch Logs | 90日 |
| 操作ログ | 社内担当者の操作（operation_logs） | RDS | 1年 |
| DBスロークエリログ | 実行時間の長いSQL | CloudWatch Logs | 30日 |
| 認証ログ | ログイン成功・失敗 | CloudWatch Logs | 90日 |

### ログレベル定義

| レベル | 出力条件 |
|---|---|
| ERROR | 例外・予期しないエラー |
| WARN | バリデーションエラー・外部IF失敗 |
| INFO | ログイン・注文作成・ステータス変更等の主要操作 |
| DEBUG | 開発環境のみ |

---

## バックアップ

### DBバックアップ

| 項目 | 設定 |
|---|---|
| 方式 | RDS自動スナップショット |
| 取得頻度 | 毎日1回（AM 3:00 JST） |
| 保持期間 | 7日間 |
| リストア目標時間（RTO） | 2時間以内 |
| リストア目標地点（RPO） | 最大24時間前 |

### ファイルバックアップ（S3）

| 項目 | 設定 |
|---|---|
| 方式 | S3バージョニング有効化 |
| 保持期間 | 90日間 |
| 対象 | 送付リスト・サンプル文面・帳票PDF |

---

## 監視（Datadog）

### 導入コンポーネント

| コンポーネント | 用途 |
|---|---|
| Datadog Agent | EC2のメトリクス収集（CPU・メモリ・ディスク） |
| APM | アプリケーションのトレーシング・レスポンスタイム計測 |
| Log Management | CloudWatch LogsからDatadogへ転送 |
| Synthetics | 外形監視（ログイン画面・注文画面のURLを定期疎通確認） |

### アラート設定

| 監視項目 | 閾値 | 通知先 |
|---|---|---|
| CPU使用率 | 80%以上が5分継続 | Slackアラートチャンネル |
| メモリ使用率 | 85%以上 | Slackアラートチャンネル |
| DBコネクション数 | 上限の80% | Slackアラートチャンネル |
| 5xxエラー率 | 1%以上 | Slackアラートチャンネル + メール |
| 外形監視失敗 | 2回連続失敗 | Slackアラートチャンネル + メール |
| ディスク使用率 | 80%以上 | Slackアラートチャンネル |

### ダッシュボード構成

| ダッシュボード | 主要メトリクス |
|---|---|
| インフラ | CPU・メモリ・ネットワーク・ディスク |
| アプリケーション | レスポンスタイム・エラー率・スループット |
| ビジネス | 注文数・ログイン数・差戻し数（日次） |

---

## インシデント対応フロー

```
アラート発火
  ↓
Slackへ通知（担当者確認）
  ↓
影響範囲の確認
  ├─ 軽微 → ログ確認・原因調査・修正
  └─ 重大 → メンテナンスモード切替 → 原因調査 → 修正 → 解除
  ↓
事後レポート作成（重大障害の場合）
```
